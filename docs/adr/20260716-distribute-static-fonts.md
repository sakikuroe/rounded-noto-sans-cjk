# ADR-20260716: 配布物は静的フォントとし、ROND 可変軸の経路はライブラリ API として保持する

## 本決定がいまどの段階にあるか (Status)

Accepted (2026-07-16)

## 本決定が必要になった背景と状況

本ツールの当初の構想は、丸みを `ROND` 可変軸として連続的に調整できる OpenType/CFF2 可変フォントへの変換である (Issue #1)。可変フォントを組み立てる実装 (`variable_font::build_variable_font` と、それを呼び出す `lib.rs` の `convert`) は完成しているが、可変フォントとしての表示品質や環境ごとの互換性の検証は完了していない。一方、丸みを固定した静的フォントを生成する経路 (`convert_static` 系) は検証済みであり、配布物として実用に足る品質に達している。初回リリースに向けて、どの経路の成果物を配布するかを決める必要がある。

## 検討した選択肢とそれぞれの長所・短所

### 静的フォントのみを配布し、可変軸の経路はライブラリ API として保持する案

- 長所: 検証済みの経路だけを利用者に提供でき、静的 OTF はあらゆる環境で安定して動作する。可変軸の実装を捨てないため、検証が済み次第公開できる。
- 短所: 到達不能な公開 API (`convert`) がリポジトリに残り、コードレビューで未使用と指摘され得る。丸みの度合いを変えたい利用者は自分でパラメータを変えて再生成する必要がある。

### ROND 可変軸を持つ可変フォントを配布する案

- 長所: 利用者が 1 つのフォントで丸みを自由に調整でき、当初の構想をそのまま実現できる。
- 短所: CFF2 可変フォントの `ROND` のような独自軸はアプリケーションの対応状況にばらつきがあり、表示品質の検証が完了していない。問題があった場合の影響が配布物全体に及ぶ。

### 何もしない (配布物を定めずリポジトリのみ公開する)

- 短所: 利用者が各自でビルドする必要があり、数 GB のメモリと外部ツールを要するビルドの負担を考えると、配布物がないことはプロジェクトの目的 (使えるフォントを届けること) に反する。

## 最終的に採用する方針

初回リリースの配布物は `convert_static` 系の経路で生成した静的フォント 4 本とする。ROND 可変軸の経路 (`convert`) は公開ライブラリ API として保持し、CLI からは公開しない。

## その方針を採用し他を退けた理由

配布物には検証済みの品質が必要であり、可変フォント経路は検証未了である。かといって実装を削除すると、当初の構想 (Issue #1) の中核を捨てることになり、検証後の公開時に作り直しが必要になる。保持に伴う「未使用の公開 API」という指摘には、`convert` の doc コメントに意図を明記することで対応した。

## 本決定によって生じる影響と後続の作業

- `convert` は到達不能な公開 API として残る。意図は doc コメントに記録済みであり、削除してはならない。
- 可変フォント出力の品質検証と CLI への公開を、今後の課題として別 Issue で扱う。
- トレードオフ: 丸みの度合いを変えるには再ビルドが必要であり、利用者の自由度は可変フォント配布に劣る。
