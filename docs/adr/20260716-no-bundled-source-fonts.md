# ADR-20260716: ソースフォントをリポジトリに同梱せず、利用者が取得する方式とする

## 本決定がいまどの段階にあるか (Status)

Proposed (2026-07-16)

## 本決定が必要になった背景と状況

フォントのビルドには、変換元となるソースフォント (Noto Sans CJK JP の各ウェイト、Noto Sans Mono CJK JP、Source Code Pro) が必要である。これらは合計で数十 MB から百 MB 超のバイナリであり、いずれも OFL 1.1 で公開元から配布されている。本リポジトリ自体のライセンスはソースコードのみに適用される MIT である。ビルドに必要なソースフォントをどう入手させるかを決める必要がある。

## 検討した選択肢とそれぞれの長所・短所

### リポジトリには同梱せず、入手と前処理の手順を README に記載する案

- 長所: リポジトリが軽量に保たれ、MIT (コード) と OFL (フォント) のライセンスの境界が明確になる。ソースフォントは常に公開元の正規の配布物から取得される。
- 短所: 利用者がビルド前にダウンロードと前処理 (インスタンス化・輪郭反転) を行う手間が生じ、公開元の URL 変更で手順が壊れる可能性がある。

### ソースフォントをリポジトリに同梱する案

- 長所: クローンするだけでビルドでき、手順の再現性が最も高い。OFL 1.1 は再配布自体を許容している。
- 短所: 百 MB 規模のバイナリで履歴が肥大化する (Git LFS を使えば緩和できるが運用コストが加わる)。第三者フォントの再配布物を含むことで、リポジトリのライセンス表記が複雑になる。

### 何もしない (入手方法を文書化しない)

- 短所: `fonts.toml` が期待する配置を利用者が推測できず、ビルドがまず成功しない。選択肢にならない。

## 最終的に採用する方針

ソースフォントはリポジトリに同梱せず、公開元からの入手と前処理の手順を README に記載する。配置先の `fonts/` は `.gitignore` に登録し、誤ってコミットされないようにする。

## その方針を採用し他を退けた理由

ライセンスの境界の明確さとリポジトリの軽さを、クローン後すぐビルドできる利便性より優先した。入手の手間は、README にコピーして実行できる形の完全な手順 (ダウンロード URL・展開・インスタンス化・輪郭反転) を記載することで補える。手順が実際に機能することは、クリーンな状態からのビルドで確認済みである。

## 本決定によって生じる影響と後続の作業

- README のビルド手順が公開元のリリース URL に依存するため、リンク切れが起きた場合は手順の更新が必要になる。
- CI ではフォントの実ビルドを行わず、コードの build・test・lint のみを検証する。フォントを含む検証を CI 化する場合は、この決定の再検討が必要になる。
- トレードオフ: 初回ビルドの所要時間と手数が増える。
