# ADR-20260716: 等幅フォントの ASCII 範囲を Source Code Pro の輪郭に差し替える

## 本決定がいまどの段階にあるか (Status)

Proposed (2026-07-16)

## 本決定が必要になった背景と状況

等幅の生成フォントの変換元である Noto Sans Mono CJK JP は、Regular と Bold の 2 ウェイトしか提供されていない。日本語グリフは丸め処理で意図した印象になる一方、欧文 (ASCII) グリフは丸めるだけでは日本語部分と太さ・造形の印象が揃わず、特に Bold 側で差が目立つ。コーディング用途では ASCII の表示品質が体験を大きく左右するため、欧文部分の品質を確保する方法を決める必要がある。

## 検討した選択肢とそれぞれの長所・短所

### ASCII 範囲を別の等幅フォントの輪郭に差し替える案

- 長所: 欧文専用に設計された高品質なグリフを使え、丸めパラメータも日本語部分と独立に調整できる。差し替え元に可変フォントがあれば、任意のウェイトをインスタンス化して日本語部分と太さを揃えられる。
- 短所: 変換パイプラインに差し替え処理 (`convert_static_with_ascii`) が加わり複雑になる。差し替え元フォントのライセンスと RFN の制約を受ける。

### 何もしない (Noto Sans Mono CJK JP の欧文をそのまま丸める)

- 長所: 実装が単純で、ライセンスの考慮対象も 1 フォントで済む。
- 短所: 2 ウェイトしかない欧文グリフの印象を丸めパラメータだけでは日本語部分と揃えられず、配布物の品質目標を満たせない。

## 最終的に採用する方針

等幅フォントの ASCII 範囲 (U+0020〜U+007E) を Source Code Pro の輪郭に差し替え、日本語部分とは独立した丸めパラメータを適用する。Regular には可変フォントからウェイト 480 をインスタンス化したもの、Bold には静的な Bold を用いる。

## その方針を採用し他を退けた理由

差し替えなしでは品質目標を満たせないことが決定的であり、複雑さの増加は差し替えロジックを `convert_static_with_ascii` に隔離することで許容範囲に収まる。差し替え元には Source Code Pro のほか IBM Plex Mono・Hack・Monaspace Neon も試作して比較し、最終的な見た目の評価で Source Code Pro を選定した。Source Code Pro は可変フォントが公開されているためウェイトを 10 単位で微調整でき (Regular は 480 を採用)、ライセンスも変換元と同じ OFL 1.1 のため混植したフォント全体を単一のライセンスで配布できる。

## 本決定によって生じる影響と後続の作業

- 差し替え元フォントの入手と前処理 (ウェイト 480 のインスタンス化) がビルド手順に加わる (README に記載済み)。
- "Source" は Adobe の Reserved Font Name のため、生成フォントの名前に使用できない (命名は ADR-20260716-derived-font-naming で決定)。
- 生成フォントの著作権表示には Source Code Pro の原著作権表示も引用する (`src/bin/generate.rs` の `build_copyright`)。
- トレードオフ: ASCII と日本語で書体の出自が異なるため、混植の境界 (例: 全角と半角の記号) で造形の一貫性に注意が必要になる。丸めパラメータの調整で吸収する。
