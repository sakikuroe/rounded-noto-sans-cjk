# ADR-20260716: 設計判断を ADR として記録する運用を開始する

## 本決定がいまどの段階にあるか (Status)

Accepted (2026-07-16)

## 本決定が必要になった背景と状況

本リポジトリでは、生成フォントの命名やフォント構成など、将来の変更を縛る設計判断がすでに複数下されている。これまでその根拠は README のライセンス節、PR 本文、コードコメントに分散して記録されており、どこが正本なのかが定まっていない。今後もパラメータ調整や配布形態の変更のたびに同種の判断が積み重なる見込みであり、判断の経緯を一定の形式で残す場所を決める必要がある。

## 検討した選択肢とそれぞれの長所・短所

### Nygard 形式の ADR を `docs/adr/` に置く案

- 長所: 1 決定 1 ファイルで置き換え (supersede) の単位が明確になり、リポジトリ内で決定の履歴を Git とともに追跡できる。
- 短所: 決定のたびに文書を書く手間が生じる。

### 何もしない (README・PR・コードコメントへの分散記録を続ける)

- 長所: 追加の作業が発生しない。
- 短所: 同じ事実が複数の場所に書かれて矛盾しやすく、PR に書いた判断は凍結されるため後から現在有効な決定を判別できない。

## 最終的に採用する方針

将来の実装を縛る設計判断を、`docs/adr/YYYYMMDD-<短いタイトル>.md` の Nygard 形式で記録する運用を開始する。

## その方針を採用し他を退けた理由

本リポジトリの設計判断には、ライセンスと商標に関わるフォント命名のように、覆すと利用者へ広く影響が及ぶものが含まれる。決定の根拠を 1 決定 1 ファイルの正本として残しておくことで、見直しの際に同じ調査と議論を繰り返さずに済む。分散記録の継続は、すでに正本が定まらない問題を起こしているため退けた。

## 本決定によって生じる影響と後続の作業

- 既存の主要な設計判断を ADR として記録する (本日付の各 ADR で対応)。
- 以後、PR に書いた判断理由のうち将来を縛るものは ADR へ昇格させ、PR からリンクする。
- トレードオフ: 決定のたびに文書化の手間が生じるが、1 ページ以内に収めることで負担を抑える。
